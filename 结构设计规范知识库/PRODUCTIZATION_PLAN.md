# 结构设计规范知识库产品化实施方案

本文档用于固化项目从“可运行 RAG 原型”升级为“成熟知识库产品”的实施路线。目标不是单纯增加模型能力，而是建立可维护、可部署、可评估、可追溯的专业规范问答系统。

## 一、产品定位

面向结构设计人员、审图人员和工程技术管理人员，提供结构设计规范的智能检索、条文问答、来源追溯和规范版本管理能力。

成熟产品应满足以下标准：

- **可稳定部署**：支持 Docker 或宿主机标准化部署，配置清晰，启动失败可定位。
- **可持续更新**：新增、替换、废止规范文档后，可稳定重建知识库。
- **回答可信**：回答必须包含规范名称、条文号、页码、原文依据或引用摘要。
- **检索准确**：支持条文号、关键词、自然语言问题和多规范对比。
- **使用安全**：API Key、跨域、限流、请求大小、日志脱敏均有明确边界。
- **可量化评估**：有标准问题集和检索/回答质量指标，避免只凭主观感觉调参。

## 二、成熟度目标

当前项目已具备 PDF 处理、ChromaDB、ZhipuAI Embedding、BM25 混合检索、MiMo 多模态回答、OpenAI-compatible API 和图片页引用能力。下一阶段重点是补齐工程化能力。

| 维度 | 当前状态 | 产品化目标 |
| --- | --- | --- |
| 工程结构 | 核心逻辑集中在 `src/main.py` | API、配置、检索、RAG、LLM、pipeline 分层 |
| 数据构建 | 可运行脚本 | 可重复构建、可追踪 manifest、可检查状态 |
| 检索质量 | 向量 + BM25 + 条文号匹配 | 标准化召回、融合、rerank、引用整理 |
| 服务稳定性 | 本地可运行 | 健康检查、错误标准化、日志观测、部署说明 |
| 安全边界 | 本地工具形态 | API 鉴权、CORS 白名单、限流、日志脱敏 |
| 测试评估 | 缺少系统测试 | 单元测试、API 测试、RAG 评估集 |
| 产品体验 | 主要通过兼容客户端使用 | Web UI、来源侧栏、文档管理、构建状态 |

## 三、实施阶段

### 阶段一：工程治理与项目清理

**目标**：让项目干净、可维护、可交接。

实施任务：

1. 清理仓库运行产物：
   - 移除 `venv/`
   - 移除 `__pycache__/`
   - 移除 `.DS_Store`
   - 移除 `logs/*.log`
   - 评估是否保留或迁移 `db/`、`data/images/` 等生成产物
2. 完善 `.gitignore`：
   - Python 缓存和虚拟环境
   - 本地数据库和日志
   - 临时图片、构建产物
   - `.env` 和本地密钥文件
3. 拆分 `src/main.py` 职责：

```text
src/
  app/
    main.py
    api/
      chat.py
      health.py
    core/
      config.py
      logging.py
      errors.py
    retrieval/
      vector_store.py
      bm25.py
      hybrid_search.py
    rag/
      service.py
      prompt.py
      citations.py
    llm/
      client.py
      stream.py
    schemas/
      chat.py
      health.py
  pipeline/
    process_documents.py
    step_04_load_to_db.py
```

4. 集中配置管理：
   - `ZHIPUAI_API_KEY`
   - `MIMO_API_KEY`
   - `MIMO_BASE_URL`
   - `MIMO_MODEL`
   - `RAG_TOP_K`
   - `RAG_MIN_SCORE`
   - `DB_DIR`
   - `IMG_DIR`
   - `IMG_BASE_URL`
   - CORS、鉴权、日志等级

阶段验收：

- 工作区不再包含应忽略的运行产物。
- `src/main.py` 只保留应用创建和路由注册。
- `.env.example` 能完整表达运行所需配置。
- README 能说明本地启动、构建知识库和调用 API。

### 阶段二：知识库构建流程产品化

**目标**：让 PDF 到知识库的过程可靠、可重复、可追踪。

**状态**：已完成统一 pipeline CLI、元数据覆盖、标准化 chunk、manifest 生成和兼容入库入口。

实施任务：

1. 建立规范元数据模型：

```json
{
  "code": "GB 50010-2010",
  "name": "混凝土结构设计规范",
  "version": "2015版",
  "source_file": "GB 50010-2010_混凝土结构设计规范_2015版.pdf",
  "effective_date": "",
  "status": "active"
}
```

2. 优化 chunk 元数据：
   - 规范编号
   - 规范名称
   - 版本
   - 章
   - 节
   - 条文号
   - 页码
   - 来源文件
   - chunk id
   - chunk 类型：正文、表格、条文说明、图片页 OCR 等
3. 增加构建命令：

```bash
python -m src.pipeline build
python -m src.pipeline rebuild --source data/raw
python -m src.pipeline status
```

4. 生成构建 manifest：
   - 文档列表
   - chunk 数量
   - embedding 模型
   - 图片数量
   - 构建时间
   - 构建参数
   - 数据版本 hash

阶段验收：

- 同一批 PDF 可以重复构建出可追踪的知识库版本。
- 每条检索结果都能追溯到规范、条文和页码。
- 新增规范不需要手工改主服务代码。

### 阶段三：检索与回答质量提升

**目标**：让系统从“能回答”升级为“答得准、可证明”。

**状态**：已完成标准化检索结果、查询解析、可插拔 reranker 默认实现、结构化 RAG 上下文和轻量评估 CLI。

标准 RAG 流程：

```text
用户问题
  -> 查询规范化
  -> 条文号精确匹配
  -> BM25 关键词召回
  -> 向量召回
  -> 结果融合
  -> rerank
  -> 引用整理
  -> LLM 生成
  -> 答案校验
```

实施任务：

1. 条文号识别增强：
   - `GB50010 第 8.2.1 条`
   - `8.2.1 是什么`
   - `混规 8.2.1`
   - `抗震规范第 6.1.3 条`
2. 检索结果融合标准化：
   - 保留 dense score
   - 保留 BM25 score
   - 保留条文号命中标记
   - 输出统一 ranking reason
3. 引入 reranker：
   - 优先评估中文 reranker
   - 支持本地模型和 API 两种实现
   - 保留无 reranker 的降级路径
4. 强制答案格式：
   - 结论
   - 依据条文
   - 来源规范
   - 页码
   - 不确定性说明
5. 建立评估集：
   - 条文查询
   - 参数解释
   - 构造要求
   - 表格查询
   - 多规范对比
   - 易混淆问题

阶段验收：

- 至少 50 个标准问题可自动评估。
- 每次检索策略调整后能比较命中率和回答质量。
- 回答缺少可靠依据时，应明确说明“不确定”或“未找到明确依据”。

### 阶段四：API 服务成熟化

**目标**：让服务可以作为内部系统稳定运行。

**状态**：已完成标准错误结构、request id、`/ready`、JSON `/metrics`、API Key 鉴权、请求大小限制、进程内限流和 CORS 默认值收紧。

实施任务：

1. 增加基础端点：
   - `/health`：进程存活
   - `/ready`：依赖就绪，包括 ChromaDB、embedding、LLM 配置
   - `/metrics`：请求量、耗时、错误率
2. 标准化错误响应：

```json
{
  "error": {
    "code": "KNOWLEDGE_BASE_NOT_READY",
    "message": "知识库尚未加载完成",
    "request_id": "..."
  }
}
```

3. 安全控制：
   - API Key 鉴权
   - CORS 白名单
   - 请求大小限制
   - 简单限流
   - 日志脱敏
4. 日志观测：
   - request id
   - 查询文本长度
   - 检索耗时
   - LLM 耗时
   - 命中 chunk 数
   - 错误原因
5. 部署整理：
   - Dockerfile
   - docker-compose.yml
   - 数据卷说明
   - `.env.example`
   - 启动检查脚本

阶段验收：

- 服务启动、依赖异常、请求异常均可定位。
- 生产配置不会默认使用全开放 CORS 或空鉴权。
- 可以通过标准命令在新机器复现运行环境。

### 阶段五：产品体验建设

**目标**：从 API 工具升级为面向设计人员的知识库产品。

**状态**：已采用 Open WebUI 作为主聊天入口，项目内置静态控制台负责知识库状态、文档清单、评估集状态、轻量问答测试和来源图片预览。

实施任务：

1. Web UI：
   - 问答窗口
   - 来源条文侧栏
   - PDF 页图片预览
   - 引用复制
   - 历史记录
2. 管理后台：
   - 文档上传
   - 构建进度
   - 知识库状态
   - 文档版本
   - 失败日志
   - 评估集运行结果
3. 专业功能：
   - 条文收藏
   - 多规范对比
   - 规范目录浏览
   - 限定某本规范检索
   - 导出问答报告

阶段验收：

- 非开发用户可以通过浏览器完成查询。
- 管理员可以查看知识库版本、构建状态和失败原因。
- 问答结果能一键定位到来源页和条文。

## 六、关键原则

产品化过程中必须坚持以下原则：

- 回答必须可追溯，不能只给自然语言结论。
- 检索必须可评估，不能只依靠人工试问。
- 知识库必须可重建，不能依赖一次性的本地状态。
- 服务必须可部署，不能只依赖开发机环境。
- 错误必须可定位，不能只返回模型生成失败。
